Verbos HTTP, CRUD, REST e RESTful

Nome: Leonardo Gabriel Russo dos Santos  |  Professor: Sandro

1. O que são verbos HTTP e qual a relação com o acrônimo CRUD?

Os verbos HTTP são métodos de requisição que indicam a ação que um cliente (por meio de um navegador ou aplicação) deseja realizar em um servidor para manipular recursos, como páginas, imagens ou dados. Esses métodos funcionam como "palavras de ação" da internet, permitindo uma comunicação padronizada entre sistemas. Eles são definidos pelo protocolo HTTP/1.1, especificado na RFC 7231.

O acrônimo CRUD representa as quatro operações básicas de gerenciamento de dados em um sistema: Create (Criar), Read (Ler), Update (Atualizar) e Delete (Excluir). Em APIs e serviços web, os verbos HTTP normalmente são utilizados para representar essas operações.

2. Quais são os verbos mais utilizados e quais são suas funções?

Os verbos HTTP mais utilizados são GET, POST, PUT, PATCH e DELETE. Eles são utilizados para indicar qual ação será realizada sobre um recurso em um servidor.

A tabela abaixo mostra a relação entre as operações CRUD e os métodos HTTP utilizados para cada uma delas.

Operação CRUDMétodo HTTPFunçãoExemplo
Create (Criar) POST Cria um novo recurso no servidor POST /usuarios
Read (Ler) GET Consulta ou obtém dados do servidor GET /usuarios
Update (Atualizar) PUT / PATCH Atualiza um recurso existente PUT /usuarios/{cpf}
Delete (Excluir) DELETE Remove um recurso do servidor DELETE /usuarios/{cpf}

Outros métodos auxiliares

Além dos métodos principais utilizados no CRUD, existem outros métodos HTTP que possuem finalidades mais específicas:

  • HEAD: Similar ao GET, mas retorna apenas os cabeçalhos da resposta, sem o corpo do conteúdo. É útil para verificar informações sobre um recurso antes de fazer o download completo.
  • OPTIONS: Retorna quais métodos HTTP são permitidos para uma determinada URL. É muito utilizado em políticas de segurança como CORS.
  • TRACE: Utilizado principalmente para testes de diagnóstico, permitindo verificar o caminho que a requisição percorreu até chegar ao servidor.

3. Qual a diferença entre REST e RESTful?

REST (Representational State Transfer) é um conjunto de princípios arquiteturais utilizados para criar serviços web. Esses princípios foram definidos por Roy Fielding em sua tese de doutorado.

Para que uma API seja considerada RESTful, ela deve seguir alguns princípios, como:

  • Cliente-Servidor: separa a interface do cliente da lógica do servidor.
  • Stateless (sem estado): cada requisição deve conter todas as informações necessárias para ser compreendida.
  • Cacheável: as respostas podem ser armazenadas em cache para melhorar o desempenho.
  • Interface uniforme: define uma forma padronizada de comunicação utilizando URLs, métodos HTTP e formatos como JSON ou XML.
  • Sistema em camadas: permite a utilização de intermediários como proxies e balanceadores de carga.
  • Código sob demanda (opcional): o servidor pode enviar código executável para o cliente, como scripts.

De forma resumida, REST é o modelo ou conjunto de princípios, enquanto RESTful é o termo utilizado para descrever APIs que realmente aplicam esses princípios na prática. A API de usuários desenvolvida neste projeto é um exemplo de API RESTful: ela utiliza os verbos HTTP corretamente (GET, POST, PUT, DELETE), identifica cada recurso por uma URL única (/usuarios/{cpf}), não armazena estado entre as requisições e retorna os dados no formato JSON.

Para demonstrar na prática o funcionamento de uma API RESTful, foi desenvolvida uma aplicação completa utilizando as seguintes tecnologias:

  • Java 21: linguagem de programação utilizada no desenvolvimento do back-end.
  • Spring Boot 4: framework que facilita a criação de aplicações Java, fornecendo configuração automática e servidor embutido.
  • Spring Data JPA: responsável pela comunicação com o banco de dados, mapeando os objetos Java para tabelas.
  • H2 Database: banco de dados relacional em memória, utilizado para armazenar os dados dos usuários durante a execução da aplicação.
  • Springdoc OpenAPI (Swagger UI): ferramenta que gera automaticamente a documentação interativa da API.
  • HTML, CSS e JavaScript: utilizados para criar a interface web que permite testar os endpoints da API diretamente pelo navegador.

Como o protocolo HTTP funciona na aplicação?

O protocolo HTTP (HyperText Transfer Protocol) é o protocolo de comunicação utilizado entre o cliente (navegador) e o servidor da aplicação. Toda interação na API segue o modelo requisição → resposta: o cliente envia uma requisição HTTP com um verbo, uma URL e opcionalmente um corpo em JSON, e o servidor processa e devolve uma resposta com um código de status e os dados solicitados.

Na aplicação desenvolvida, o fluxo funciona da seguinte forma:

Ação no navegadorRequisição HTTP enviadaO que o servidor fazResposta
Clicar em GET GET /usuarios Busca todos os usuários no banco de dados 200 OK + lista em JSON
Clicar em POST POST /usuarios + corpo JSON Valida os dados e salva o novo usuário 201 Created + usuário criado
Clicar em PUT PUT /usuarios/{cpf} + corpo JSON Localiza o usuário pelo CPF e atualiza os dados 200 OK + usuário atualizado
Clicar em DELETE DELETE /usuarios/{cpf} Localiza o usuário pelo CPF e remove do banco 204 No Content
CPF não encontrado GET / PUT / DELETE com CPF inexistente Lança exceção de usuário não encontrado 404 Not Found + mensagem de erro
CPF já cadastrado POST /usuarios com CPF duplicado Lança exceção de CPF já existente 409 Conflict + mensagem de erro

Os códigos de status HTTP são parte fundamental do protocolo: eles indicam ao cliente o resultado da requisição de forma padronizada. Códigos 2xx indicam sucesso, 4xx indicam erros causados pelo cliente (dados inválidos, recurso não encontrado) e 5xx indicam erros internos do servidor.
